Section 00

시작하기 전에

Hermes Agent를 처음 들어본 분 — Claude Code/Cursor와 뭐가 다른지, 지금 깔아도 되는지 5분 안에 결정합니다.

처음 'Hermes Agent'라는 이름을 들으면 이런 의문이 떠오릅니다. "또 새로운 AI 에이전트야?", "Claude Code 잘 쓰고 있는데 굳이?", "오픈소스라는데 결국 GPU 사야 되는 거 아냐?". 이 가이드는 그 세 가지 질문에 먼저 답하면서 시작합니다.

💡

한 줄 요약: Hermes Agent는 코딩 어시스턴트가 아니라 '내 서버에 상주하면서 쓸수록 똑똑해지는 자율 비서'입니다. Claude Code의 대체가 아니라 보완재로 보는 게 맞습니다.

한국 사용자가 자주 내리는 결론: "코드 짜는 건 Claude Code / Cursor가 낫고, 매일 돌아가는 자동화 작업·리서치·메신저 자동응답은 Hermes가 낫다. 둘 다 쓴다." 시니어 AI 아키텍트 Sathish Raju도 "코드 생성에선 Cursor·Claude Code가 우월하다"고 솔직히 인정합니다.[20][24][26]

Q2. 비개발자도 쓸 수 있나요?

가능은 합니다. 단, 터미널을 한 번도 안 열어본 분이라면 셋업에 평균 40분 정도 잡아야 합니다. "40분의 설정 과정이 모든 사용자에게 친화적이지는 않다"는 건 6주 사용 후기에서도 언급된 진입 장벽입니다.[21]

⚠️

Windows를 쓰신다면: 네이티브 Windows는 공식 미지원입니다. WSL2(Windows Subsystem for Linux)를 먼저 깔고 그 안에서 진행해야 합니다. 이 단계에서 막히는 분이 많아요.

Q3. 결국 GPU 사야 하나요?

아니요. 모델은 클라우드(OpenRouter·OpenAI·Anthropic 등)를 쓰면 GPU 없이 OK입니다. 공식 사용 사례에는 $5~$20짜리 VPS, Raspberry Pi 4에서 돌리는 사례도 있습니다. 로컬 GPU는 옵션이지 필수가 아닙니다.[6]

이런 분에게 맞습니다

  • 매일 돌아가는 자동화(SEO 리포트, 모니터링, 데일리 요약 등)를 메신저로 받고 싶은 분
  • 여러 LLM을 비용·성능 보고 갈아끼우고 싶은 분
  • 1인 사이드 프로젝트 운영자 — "디자이너에게도 게임체인저"라는 평이 있을 만큼 비개발 직군에서도 활용 사례가 많습니다
  • OpenClaw에서 갈아타려는 분 — 단, 자기학습 기능이 기본 비활성이라 그대로 쓰면 별 차이 없음 (§13 한계 참조)

이런 분에게는 맞지 않습니다

  • 코드 생성 / 페어 프로그래밍이 주 목적인 분 → Claude Code · Cursor가 낫습니다
  • 프로덕션 안정성이 최우선인 엔터프라이즈 → 출시 10주 차로 검증 시간 부족
  • 터미널·서버 환경에 전혀 익숙하지 않은데 Windows만 쓰는 분 → WSL2 셋업이 더 어려운 산

한 줄 요약: 코딩은 Claude Code, 운영·리서치·자동화는 Hermes — 이 분담이면 후회 없습니다. 다음 섹션에서 Hermes의 '자기학습'이 정확히 뭘 뜻하는지 봅니다.

Section 01

Hermes Agent란?

한 줄 정의 + 비유 1개 + 다른 도구와 결정적으로 다른 1가지를 5줄 안에 알려드립니다.

Hermes Agent는 '내 서버에 상주하면서, 자기가 한 일을 기억하고, 시간이 지날수록 더 잘하게 되는' 자율 AI 에이전트입니다. Nous Research가 2026년 2월에 오픈소스(MIT)로 공개했고, 출시 10주 만에 GitHub 100k stars를 돌파해 화제가 됐습니다.[1][7][27]

An autonomous agent that lives on your server, remembers what it learns, and gets more capable the longer it runs.

— Hermes Agent 공식 사이트, [1]

비유로 한 번에

다른 AI 에이전트가 "매일 처음 만나는 임시 알바"라면, Hermes는 "옆자리에 앉아서 같이 일하는 신입 동료"입니다. 어제 가르쳐 준 걸 오늘 또 가르칠 필요가 없고, 어떤 작업을 한 번 잘 해내면 "이렇게 하면 된다"는 메모를 자기 노트에 남겨둡니다 — 그게 'Skill'입니다.

결정적으로 다른 1가지: Self-Improving Learning Loop

기술적으로는 두 시스템이 연결되어 작동합니다.[2][4]

시스템역할저장 위치
Persistent Memory사실·맥락을 자동 저장 + 관련도 기반 회수 (SQLite + FTS5)~/.hermes/
Skill System한 번 잘 해낸 작업을 단계별 절차로 패키지화. 비슷한 작업 시 재호출프로파일별 스킬 라이브러리

출처: [2] [4] [5]

한 사용자는 6주 동안 명시적으로 요청하지 않았는데 약 43개의 Skill이 자동으로 쌓였다고 보고합니다. 같은 작업이 1주 차엔 20분, 4주 차엔 12분, 6주 차엔 8분으로 줄었다는 측정도 있습니다.[21][22]

ℹ️

단, 이 자기학습 기능은 기본값이 'off'입니다. 그냥 깔고 쓰면 일반 에이전트와 다를 게 없어요. 이걸 켜는 방법은 §09 설정 파일에서 다룹니다.

한 줄 요약: '기억하는 + 스스로 절차를 만드는 AI 에이전트'. 이 두 단어가 Hermes의 전부이자 그것 때문에 화제입니다.

가치 인지 — 뭘 할 수 있고, 왜 지금인가

Section 02

이걸로 뭘 할 수 있어

활용 사례 6가지 — 코딩 보조 너머의 '운영·자동화'에 진짜 가치가 있다는 걸 보여드립니다.

Hermes의 진짜 매력은 "코드 한 줄 짜주는 거"가 아니라 "내가 자는 동안에도 돌아가는 비서"입니다. 메신저 19종, 무인 스케줄링, 200+ LLM 모두 이 목적을 위한 부품입니다.[2][7]

활용 사례 6가지

  1. 개인 SEO 리포터: 매일 오전 8시에 경쟁사 블로그·검색 트렌드 수집 → Telegram으로 요약 전송. 6주 후기에서 가장 많이 언급된 시나리오입니다.
  2. 서버 모니터링 + 자가 복구 메모리: 에러가 발생하면 로그 분석 → 해결법을 Skill로 저장 → 같은 에러 재발 시 즉시 처리. 한 번 겪은 문제를 두 번 안 겪는 구조.
  3. 리서치 어시스턴트: 4개 플랫폼(Reddit/HN/X/뉴스) 동시 검색 → 한 번에 정리. 단, 자가평가 결함 때문에 결과 검증은 사람이 직접 해야 합니다(§13).
  4. 메신저 통한 비동기 협업: Slack 스레드별로 별도 세션 유지. 디스코드 채널마다 다른 페르소나. WhatsApp/Signal로 외부 클라이언트 응대.
  5. Cron 자동화: '매주 월요일 오전 9시에 X를 해줘'를 자연어로 등록. 결과는 Telegram·이메일로 자동 전달.
  6. OpenWebUI / LM Studio 백엔드: OpenAI 호환 API(/v1)를 노출해 외부 UI에서 Hermes를 모델처럼 호출. 사내 챗봇의 두뇌로 쓰는 패턴.

출처: [2] [7] [9] [14] [21] [22]

Before / After: 매일 SEO 리포트

상황: 경쟁사 블로그 신규 글을 매일 모아 요약

기존 방식 (45분/일)

  1. 수동으로 5개 블로그 RSS 확인
  2. 구글 트렌드에서 키워드 변동 체크
  3. 노션에 정리 + 팀 슬랙 공유

Hermes 방식 (8분, 자동)

$ hermes
> 매일 오전 8시에 경쟁사 블로그 5개 신규 글 요약하고
  변동 키워드 정리해서 텔레그램으로 보내줘.
  두 번째 실행부터는 어제 다룬 주제는 빼고.
ℹ️

두 번째 실행부터 "어제 다룬 주제는 빼고"가 가능한 이유 = persistent memory. 이게 Hermes의 본질이에요.

한 줄 요약: 코드 보조보다 '운영 자동화'에 강합니다. "자는 동안에도 일하는 비서"가 가장 적합한 사용처.

Section 03

왜 지금 트렌드인가

감이 아니라 숫자로 — 출시 10주 만의 변화를 그래프 없이도 한눈에.

시그널수치변화 / 의미
GitHub Stars (현재)~130k2026-02 출시 → 10주 만에 100k 돌파, Claude Code 추월
v0.8.0 단일일 신규 stars+6,400릴리스 당일 폭증
릴리스 빈도5주 7회 메이저, 6주 차 12회v0.12.0 (2026-04-30 'Curator')까지 빠른 회전
Open Issues~2,900스타 증가에 비례해 이슈 폭증 → 성숙도 미달 신호이기도
HN 토론 게시글최근 4주 3건+기술 커뮤니티 확실한 관심
한국 커뮤니티Threads · Brunch · OPSOAI 등 4월 다수한 달간 한국어 후기 본격 등장

출처: [7] [17] [18] [19] [23] [24] [25] [27]

트렌드 시그널은 두 갈래의 동시 흐름입니다. 한쪽엔 "OpenClaw 보안 이슈와 세션 망각 피로"가 있고, 반대쪽엔 "기억하는 + 스스로 진화하는" Hermes의 차별점이 있습니다. 둘이 맞물려 빠르게 화제가 됐습니다.[17][24][27]

⚠️

단, 화제성에 거품이 섞여 있다는 비판도 동시에 등장합니다. Reddit에는 "신규 계정의 템플릿 글이 다수"라는 astroturfing 의혹 댓글이 +30 upvote를 받고 있고, 36kr는 'NOUS 토큰' 관련 Web3 연계 의혹을 경계 신호로 짚었습니다. 이 가이드는 호평·혹평·광고를 분리해서 보여드립니다.

한 줄 결론

"기억하는 + 스스로 절차를 만드는"이라는 차별점이 OpenClaw의 한계를 정확히 찌르면서, 출시 10주 만에 영문권·한국·중화권 동시 스파이크. 단, 안정성·자가평가 결함 등 실제 한계는 §13에서 솔직히 다룹니다.[17][23][27]

한 줄 요약: '기억하는 AI'의 첫 양산 사례. 다만 화제성과 검증 시간 부족이 동시에 진행 중인 도구라는 점은 인지하고 시작합시다.

⚠️

숨은 비용 주의: 메모리 요약·스킬 자동 생성도 LLM 호출입니다. 한국 10년차 개발자 OPSOAI 리뷰는 "지속적인 LLM API 호출이 메모리 요약과 스킬 생성에 누적되어 '예상치 못한 비용'이 발생한다"고 경고합니다. 첫 한 달은 OpenRouter 한도를 낮춰두고 모니터링 권장.

시스템 요구사항

  • OS: Linux / macOS / WSL2 / Android(Termux). Native Windows 미지원.
  • Python: 3.11+ (3.13은 의존성 호환 깨짐 — 3.11 또는 3.12 권장)
  • 모델 컨텍스트: 최소 64,000 토큰. 미만이면 startup 자체가 거절됩니다.
  • 의존성: uv 패키지 매니저, Node 22+ (MCP 도구용), 선택적으로 ffmpeg/ripgrep
  • 디스크: ~/.hermes/ 누적 — 수 주 사용 후 GB 단위 가능. 주기적 정리 필요

Block 4

첫 경험 — 5분 안에 '오 됐다' 만들기

Section 05

설치 (OS별 Step-by-Step)

공식은 "5분이면 됩니다"라지만 실제는 평균 15~40분. 여기서 막히는 4가지 지점을 미리 우회합니다.

⚠️

Windows 사용자: 네이티브 미지원입니다. PowerShell(관리자)에서 wsl --install → 재부팅 → Ubuntu 터미널 안에서 아래 단계 진행하세요. WSL이 또 실패하면 BIOS에서 가상화 활성화 + Windows Features에서 WSL 켜기를 점검하세요.

Step 1. 원라이너 설치

curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash

공식 권장 경로. macOS/Linux/WSL2/Termux 자동 감지하여 ~/.local/bin/hermes 에 설치. sudo 절대 사용 금지 — 권한 꼬이면 ~/.hermes/ 통째로 지워야 합니다.

💡

GitHub raw가 차단된 사내망이라면 미리 export GITHUB_TOKEN=... 후 재시도. 그래도 안 되면 git clone 후 로컬 install.sh 실행.

Step 2. 셸 PATH 적용 (가장 흔한 막힘)

source ~/.bashrc   # bash
source ~/.zshrc    # zsh

설치 스크립트는 "성공"이라 했지만 새 터미널을 열거나 위 명령을 실행하지 않으면 hermes 명령이 안 잡힙니다. EvoMap이 "the single most common Hermes complaint"로 부르는 가장 빈번한 막힘 1순위.

💡

그래도 안 되면 ls ~/.local/bin/hermes 로 파일 존재 확인 → echo $PATH 에 ~/.local/bin 들어있는지 확인.

Step 3. doctor 진단 (선택이지만 필수)

hermes doctor

환경·경로·의존성·프로바이더 연결성 점검. EvoMap·BSWEN·HermesAtlas 모두 "이거 안 돌리고 디버깅한 시간이 아깝다"고 입을 모읍니다.

💡

에러 → 해당 항목 검색 → 거의 다 알려진 우회법 있음.

Step 4. Apple Silicon 보정 (M1/M2/M3 사용 시)

echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc

설치 스크립트가 Homebrew 경로를 자동으로 ~/.zshrc에 안 넣어주는 경우가 있어, ffmpeg·ripgrep 같은 의존을 못 찾는 사례가 보고됩니다.

💡

이 단계는 인텔 Mac · Linux · WSL에선 불필요.

💡

Python 버전 충돌(3.13에서 tiktoken pyo3 에러)이 발생하면, 명시적으로 3.11로 venv: uv venv venv --python 3.11. 자세한 우회법은 §11 자주 막히는 곳 Top 3.

한 줄 요약: curl 한 줄 → 셸 reload → hermes doctor. 이 3단계만 차분히 가면 80%는 끝. 막히면 §11으로.

Section 06

첫 실행 — 5분 안에 첫 결과물

모델 등록부터 첫 명령까지. 여기서 첫 'oh, 됐다' 순간을 만듭니다.

Step 1. 모델 등록 (인터랙티브 마법사)

hermes model

공급자 선택부터 API 키 입력까지 안내합니다. 첫 사용자에겐 OpenRouter 추천: 200+ 모델 한 키로 접근 + 64K 컨텍스트 자동 충족.

💡

API 키 invalid 에러의 90%는 클립보드 trailing whitespace입니다. 우회법은 §11 Top 2 참고.

Step 2. TUI로 실행

hermes --tui

TUI(Text UI)는 멀티라인 편집·슬래시 자동완성·세션 히스토리·툴 출력 스트리밍을 지원합니다. 클래식 CLI(hermes)보다 권장.

💡

도중에 /compress로 대화 압축, /model로 모델 전환, /reload-mcp로 MCP 서버 재로드.

Step 3. 첫 프롬프트 (공식 권장 예시)

> Summarize this repo in 5 bullets and tell me what the main entrypoint is.

현재 디렉토리를 분석해 README·구조를 5줄로 요약. "오, 진짜 내 폴더를 읽네"의 첫 순간.

Step 4. 세션 재개 검증

hermes -c   # 또는 hermes --continue

방금 끊었다 다시 들어가도 어제·오늘 대화가 이어집니다. 이게 Hermes의 본질을 처음 체감하는 순간.

💡

공식 권장 두 번째 검증 프롬프트: "What's my disk usage? Show the top 5 largest directories."

💡

OpenClaw·Claude Code 사용자라면: 셋업 후에 반드시 ~/.hermes/config.yaml에서 persistent_memory: true와 skill_generation: enabled 둘 다 켜세요. 안 켜면 일반 단발 에이전트와 차이가 없습니다 — 이게 OpenClaw에서 갈아탄 사용자가 가장 많이 후회하는 함정입니다.

한 줄 요약: hermes model → hermes --tui → 첫 프롬프트 → hermes -c로 재개. 4단계로 첫 경험 완료.

좋은 사용 vs 나쁜 사용

Hermes는 '기억하는' 에이전트라 프롬프트 작성법도 살짝 다릅니다.

❌ 나쁜 예 (단발 작업으로만 쓰기)

오늘 경쟁사 블로그 5개 요약해줘

✅ 좋은 예 (학습 의도 명시)

매일 오전 8시에 이 5개 블로그 신규 글 요약해서 텔레그램으로 보내줘. 첫 실행 후엔 '같은 작업'이라는 걸 기억하고, 두 번째부터는 어제 다룬 주제는 빼줘. 이 작업을 'daily-seo-report' Skill로 저장해.

❌ 나쁜 예 (검증 없이 결과 신뢰)

수질 데이터 수집해서 보고서 만들어줘 — 결과를 그냥 임원에게 전달

✅ 좋은 예 (자가평가 결함 인지)

수질 데이터 수집해서 보고서 만들어줘. 단, 출처 URL 모두 표기하고, 데이터가 부족하거나 추정이 들어간 부분은 '추정'으로 명시해줘. 결과 검증은 내가 직접 하겠다고 가정하고 결과만 보고할 것.

⚠️

Reddit에서 +107 upvote를 받은 비판: "It always thinks it did a good job. ALWAYS." Hermes는 자기 결과를 과대평가하는 경향이 강합니다. 데이터 정합성이 중요한 작업은 반드시 출처 명시 + 추정 표기를 프롬프트에 못박으세요.

한 줄 요약: '단발 명령'보다 '의도 + 학습 단서 + 검증 조건'을 함께 줘야 Hermes의 본질이 살아납니다.

config.yaml 핵심 옵션 (Hermes의 본질을 켜는 곳)

# ~/.hermes/config.yaml

# === Hermes의 본질을 활성화 ===
persistent_memory: true       # 기본 false! 반드시 켜기
skill_generation: enabled     # 기본 off! 반드시 켜기

model:
  provider: openrouter
  name: anthropic/claude-sonnet-4-6
  context_length: 200000      # 자동 감지 실패 시 수동 지정

terminal:
  shell_init_files:
    - ~/.zshrc
    - ~/.nvm/nvm.sh        # node/nvm 못 찾으면 추가

display:
  tool_progress: "on"      # Telegram 게이트웨이에선 "off" 권장

skills:
  platform_disabled:
    telegram: false        # Telegram 슬래시 100개 한도 회피 시 true
⚠️

가장 흔한 함정: persistent_memory와 skill_generation이 기본 비활성입니다. 이걸 켜지 않으면 OpenClaw·Claude Code와 다를 게 없어요. 새 사용자의 가장 큰 후회 1순위.

백업 & 이전

# 전체 ~/.hermes/ zip 백업
hermes backup

# 자격증명 제거된 프로파일 export (팀 공유용)
hermes profile export myprofile

문제 해결 — 현실에서 진짜 일어나는 일

Section 11

자주 막히는 곳 Top 10 Q&A

여러 블로그 교차로 확인된 진짜 막힘 10가지. 거의 다 여기 있어요.

Q1. hermes: command not found (PATH 미반영) — 빈도 1위

원인: 설치는 성공했지만 현재 셸이 PATH 변경을 모름. "single most common Hermes complaint, and it isn't really a Hermes bug, it's a shell behaviour" (EvoMap).

해결:

  1. source ~/.bashrc 또는 source ~/.zshrc
  2. 새 터미널 창 열기
  3. 그래도 안 되면 ls ~/.local/bin/hermes 로 존재 확인 → echo $PATH 점검
source ~/.zshrc

Q2. Provider error: invalid API key (실은 키 끝 공백)

원인: 클립보드 복사 시 trailing whitespace/줄바꿈. HermesAtlas는 "trailing whitespace (90% of cases)"로 단정.

해결:

  1. heredoc 입력으로 공백 자동 제거
  2. Ctrl-D로 입력 종료
  3. OpenRouter는 region/credits 잔액도 같이 확인
hermes config set model.api_key "$(cat)"
# 키 붙여넣고 Ctrl-D

Q3. Python 3.13에서 tiktoken pyo3 / pathlib 깨짐

원인: 의존성 생태계가 3.13 미준비. 공식은 3.11/3.12 권장.

해결:

  1. uv 격리 venv 정상 동작 확인 (시스템 Python이 먼저 잡혀선 안 됨)
  2. 명시적 3.11 venv: uv venv venv --python 3.11
uv venv venv --python 3.11

Q4. Windows 네이티브에서 무한 실패

원인: 공식 미지원. 모든 워크로드는 Unix userland에서만 동작.

해결:

  1. PowerShell(관리자)에서 wsl --install → 재부팅 → Ubuntu 터미널 안에서 설치
  2. WSL이 또 실패하면 BIOS 가상화 활성화 + Windows Features에서 WSL 켜기
wsl --install

Q5. GitHub 차단/프록시로 설치 스크립트 timeout

원인: ISP/방화벽이 GitHub 22번 또는 raw 도메인 차단. "GitHub's SSH port (22) may be blocked by ISPs or firewalls".

해결:

  1. HTTPS clone path로 전환, 프록시 환경변수 정렬
  2. Git 미러 사용 또는 수동 clone 후 로컬 설치
  3. export GITHUB_TOKEN=... 후 재시도
export GITHUB_TOKEN=ghp_xxx
curl -fsSL https://... | bash

Q6. Gateway는 떴는데 Telegram 봇 무응답 (사일런트)

원인: 봇 토큰 오타 또는 GATEWAY_HEARTBEAT 미설정으로 크래시가 가려짐.

해결:

  1. ~/.hermes/logs/gateway.log 먼저 확인
  2. .env에 GATEWAY_HEARTBEAT=true 추가 → 크래시 시 IM에 'service offline' 자동 통지
  3. @BotFather에서 토큰 재발급 후 hermes gateway setup 재실행
echo 'GATEWAY_HEARTBEAT=true' >> ~/.hermes/.env
hermes gateway run

Q7. Connection reset by peer / OpenAI-compatible endpoint

원인: Base URL 끝에 /v1 누락 (OpenAI 호환 엔드포인트 규칙).

해결:

  1. LM Studio 로컬은 server address + /v1 필수, key는 임의값 'lmstudio'
  2. context length는 비워두면 자동 감지
# config.yaml
model:
  base_url: http://localhost:1234/v1
  api_key: lmstudio

Q8. Long context 64K 미충족으로 startup 거절

원인: Hermes는 multi-step tool-calling을 위해 최소 64,000 토큰 컨텍스트 요구.

해결:

  1. OpenRouter 권장 (200+ 모델 접근)
  2. Ollama 로컬은 OLLAMA_CONTEXT_LENGTH=32768 이상 export 후 ollama serve
export OLLAMA_CONTEXT_LENGTH=65536
ollama serve

Q9. 자기학습/메모리가 발현 안 됨 — "OpenClaw 그대로네"

원인: persistent_memory, skill_generation이 기본 off.

해결:

  1. config.yaml에서 명시적 enable
  2. MEMORY가 비어있으면 "remember this"를 명시적으로 지시
# ~/.hermes/config.yaml
persistent_memory: true
skill_generation: enabled

Q10. hermes update 머지 충돌

원인: 로컬 git 수정사항이 남아있음.

해결:

  1. git stash → hermes update → git stash pop
git stash
hermes update
git stash pop

한 줄 요약: Top 1~3(PATH·API key·Python)이 80%. 그래도 막히면 hermes doctor + ~/.hermes/logs/ 두 곳.

디버깅 워크플로우 / Pro Tips

에러를 마주쳤을 때 따라갈 3단계 + 실사용 팁 8개.

에러 마주치면 3단계

  1. 1단계: hermes doctor — 환경/경로/의존/프로바이더 자동 진단. 이거 안 돌리고 디버깅한 시간이 가장 아까운 시간.

  1. API key는 heredoc으로 — hermes config set model.api_key "$(cat)" + 붙여넣기 + Ctrl-D. trailing whitespace 90% 잡힘.
  2. OpenRouter를 첫 모델로 — 200+ 모델 한 키, 64K 컨텍스트 자동 충족.
  3. /compress를 자주 — 비용 폭주 1순위 방어선.
  4. 세션 정리 주기화 — hermes sessions clean --before YYYY-MM-DD 매주.
  5. Multi-agent에선 role/memory scope를 엄격히 분리 — 안 그러면 chaos.
  6. 장시간 태스크는 stage checkpoint 명시 — context degradation 방지.
  7. WSL 경로는 표준화 — /mnt/c/...로 통일.
  8. 사용 안 하는 extras는 설치하지 말 것 — 에러 없이 기능만 빠지므로 처음엔 명시적으로 pip install "hermes-agent[mcp,messaging]".

출처: [11] [12] [13]

한 줄 요약: doctor → logs → Discord. 그리고 OpenRouter + heredoc + /compress 3종 세트는 첫날부터 습관화.

MCP로 외부 도구 연결

MCP(Model Context Protocol) 서버를 연결하면 Claude Code의 MCP와 동일한 방식으로 외부 도구 추가 가능. GitHub·Sentry·PostgreSQL 등 다 됩니다.[5]

# ~/.hermes/config.yaml
mcp_servers:
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "${GITHUB_TOKEN}"
💡

MCP 서버 변경 후 즉시 재로드: 세션 안에서 /reload-mcp. 재시작 안 해도 됩니다.

한 줄 요약: 메신저 한 줄 명령으로 비동기 비서 완성. MCP까지 붙이면 외부 SaaS도 도구화 가능.

Section 15

고급 자동화 — Cron · Subagent · OpenWebUI 연결

여기까지 오면 Hermes는 도구가 아니라 인프라가 됩니다.

Cron — 자연어 스케줄링

> 매주 월요일 오전 9시에 지난 주 GitHub PR 활동 요약해서 슬랙 #weekly로 보내줘.
  결과를 'weekly-pr-digest' Skill로 저장해.

자연어로 등록하면 Hermes가 cron 표현식으로 변환합니다. 두 번째 실행부터는 같은 Skill이 호출되어 일관성 유지.[4][7]

Subagent 위임 — 멀티에이전트 워크플로우

Hermes는 격리된 서브에이전트를 spawn할 수 있습니다. 메인 에이전트의 컨텍스트를 보호하면서 병렬 작업 가능. 단, 위임 시 role/memory scope를 엄격히 분리해야 "chaos"가 안 생깁니다.[1][12]

⚠️

서브에이전트는 부모와 컨텍스트 분리. "sub-agent context는 부모와 분리되어 있음 — 잊으면 멀티에이전트 전체가 hallucination" (EvoMap 7개 이슈 중 하나).

OpenWebUI 연결 — Hermes를 모델처럼 호출

# ~/.hermes/config.yaml
api_server:
  enabled: true
  key: my-secret-key
  port: 8642

OpenWebUI Connections에 http://localhost:8642/v1 추가 (Docker는 host.docker.internal). 이제 사내 챗봇이 Hermes를 두뇌로 쓸 수 있습니다.[9]

한 줄 요약: Cron + Subagent + OpenWebUI 3종 세트로 1인 운영 자동화 인프라 완성.
