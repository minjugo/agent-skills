# 세션별 퍼실리테이션 플레이북 (WS1~WS6)

각 세션 시작 직전에 해당 항목을 읽으세요. 구성: **목적 / 강사 강의 내용(게이트 해제 신호) / 던질 질문 / 프로브 / 품질 체크리스트 / 막힘 해소 / 환경 도메인 예시**.

> 공통 원칙: 질문은 한 번에 하나. 답을 듣고 되비추기. 막히면 예시로 물꼬. 정답 주입 금지.

---

## WS1 · 자동화하고 싶은 업무 (큰 그림)

**목적:** 무엇을 자동화하고 싶은지 큰 그림부터 정한다(출발점 정하기). 아직 순서도는 그리지 않는다.

**강사 강의 내용(이게 끝나면 활동):** 왜 큰 그림부터 그리는지(줌아웃→줌인), 기획안은 출발점 후보일 뿐 바꿔도 됨.

**던질 질문 (순서대로, 하나씩):**
1. "오늘 자동화해보고 싶은 업무가 뭐예요? 한 문장으로 편하게요. (제출한 기획안 그대로여도 좋고, 다른 업무여도 좋아요)"
2. "그 업무, 지금 뭐가 제일 답답해요? 반복돼서? 시간이 오래 걸려서? 자료가 흩어져 있어서?"
3. "이 일은 누가, 언제, 얼마나 자주 해요?"
4. "지금은 그 업무가 대충 어떤 순서로 굴러가요? 큰 흐름만 한두 문장으로요. (자세한 건 다음 단계에서)"
5. "이게 자동화되면 뭐가 제일 달라지면 좋겠어요?"

**프로브:**
- 업무가 너무 크면: "그 안에서도 가장 손이 많이 가는 한 덩어리만 고른다면요?"
- "답답함"이 추상적이면: "최근에 그 일 하면서 '아 이거 진짜…' 했던 순간 하나만요."

**품질 체크리스트:**
- [ ] 업무가 *하나*로 좁혀졌다(여러 개 뭉치 X)
- [ ] '왜'가 구체적이다(반복/시간/오류/분산 중 하나로 말할 수 있음)
- [ ] 누가·빈도가 있다

**막힘 해소:** 후보가 여럿이면 "그중 매일/매주 하는 거, 제일 지겨운 거 하나"로 좁혀준다. 아예 막히면 환경 예시를 들어 "이런 결 거예요" 보여주기.

**환경 도메인 예시:** 인허가 변경 사전검토 / CCTV 하수관로 판독 보고서 작성 / 발주 전 업체 정보 모아 확인 / 측정망 데이터 이상치 점검 / 환경 뉴스 모아 브리핑.

## WS2 · As-Is 업무 Flow

**목적:** 고른 업무가 지금 '어떻게' 굴러가는지 순서를 적고, 병목(P)·사람 필수(H)·Agent 가능 지점(◎)을 표시한다.

**강사 강의 내용:** 순서도 도형 규칙, 세분화, H(사람 필수)·P(Pain Point) 개념.

> 순서도는 `references/flowchart_rules.md`의 도형 규칙을 따른다(타원·사각형·평행사변형·마름모·화살표).

**던질 질문:**
1. "이 업무, 맨 처음 뭐가 들어오면 시작돼요? (트리거) 그리고 다 끝나면 뭐가 나와요? (최종 산출물)"
2. "시작부터 끝까지, 단계를 순서대로 말해줄래요? 한 단계씩요. (제가 받아 적을게요)"
   - 단계를 받으며 **'~하고 ~한다'면 둘로 나누자고** 제안. 한 단계 = 한 행위.
3. **분기 점검(중요):** "이 중에 **Yes/No로 갈리거나, 안 되면 앞으로 되돌아가는** 지점 있어요?" → 있으면 그 단계를 **마름모**로 바꾸고, Yes·No가 각각 어디로 가는지(복귀 포함) 함께 적는다. (예: "승인 통과?" → No면 보완 후 앞 단계로 복귀)
4. (단계가 모이면) "이 중에 제일 시간 끌거나 반복되거나 번거로운 데 — 병목(P)은 어디예요?"
5. "반대로, 사람이 *꼭* 판단·승인해야 하는 데(H)는 어디예요? (책임·규제·관계 같은)"
6. "그럼 'AI가 대신해도 되겠다' 싶은 구간(◎)은 어디예요? 여러 개여도 좋아요."

**프로브:**
- 단계가 뭉뚱그려지면: "그 단계 안에서 실제로 손으로 하는 일은 몇 개예요?"
- P를 못 찾으면: 시간·정확성·반복·정보·시스템·협업 6유형을 읽어주고 고르게.

**품질 체크리스트:**
- [ ] 시작 트리거와 최종 산출물이 있다
- [ ] 단계가 *한 단계 한 행위*로 적혀 있다(접속사로 뭉친 것 분리)
- [ ] P / H / ◎ 가 각각 최소 하나씩 표시됐다
- [ ] ◎(Agent 가능 지점)가 하나 이상 있다 — 다음 단계 To-Be의 재료

**막힘 해소:** 단계를 못 떠올리면 "어제 그 일 했다 치고, 컴퓨터 켜고 제일 먼저 뭐 했어요?"로 시간순 회상. P/H가 헷갈리면 "이거 틀리면 누가 책임져요?"(=H) 질문.

**진행 파일 기록 형식:** 단계를 `1. 단계명 [P|H|◎]` 리스트로 저장(태그는 해당될 때만).

## WS3 · To-Be(AX) Flow — Agent 로드맵

**목적:** As-Is의 ◎를 Agent 박스로 바꿔 **미래 흐름 전체**를 그린다. 만들고 싶은 Agent를 *다* 넣는다(= 로드맵). 1개로 좁히지 않는다(그건 다음 단계).

**강사 강의 내용:** To-Be를 먼저 통째로 그리는 이유, single/multi/sub 개념(흐름으로 자연 표현).

> **중요(참조 원칙):** "WS2의 2~5번"처럼 **번호로만 가리키지 않는다.** 수강생이 위로 스크롤하지 않아도 되게, 그 ◎ 구간이 *무슨 단계였는지 내용을 다시 풀어* 말한다. 예: "방금 As-Is에서 *'여러 곳에서 자료를 모아 정리하던 그 묶음'*에 '여기 AI가 도와줄 수 있겠다(◎)'고 하셨죠 — 그 부분을 Agent로 바꿔볼게요." (※ 실제 수강생 업무의 표현으로)

**던질 질문:**
1. (첫 ◎ 구간의 *내용을 풀어 말한 뒤*) "이 부분을 Agent가 맡으면, *무엇을* 해주면 좋을까요? 한 줄로요."
2. (각 ◎ 구간마다, 역시 *내용을 풀어 말하며* 반복) "그다음, *'…'* 이 부분은 Agent가 뭘 해줄까요?"
3. "이 Agent들 사이에서, 사람이 검토·승인·판단하는 지점은 어디예요? (마름모로 표시할게요)"
4. "그려놓고 보니 어때요? 만들고 싶은 Agent가 더 있으면 지금 다 넣어도 돼요 — 이게 우리 팀 'Agent 로드맵'이에요."

**프로브:**
- Agent가 1개만 나오면: "그 일 앞뒤로, 자료 모으거나 결과 정리하는 것도 Agent가 할 수 있지 않을까요?" (multi로 확장 유도, 단 강요 X)
- 한 Agent가 너무 많은 걸 하면: "그건 두 개로 나누는 게 자연스러울 것 같은데, 어때요?"

**품질 체크리스트:**
- [ ] As-Is의 ◎가 Agent 박스로 바뀌었다
- [ ] Agent가 **2개 이상** 등장하면 좋다(로드맵다움) — 단 1개여도 OK
- [ ] 각 Agent 박스에 '무엇을 하는지' 한 줄이 있다
- [ ] 사람 검토·승인 지점(마름모)이 표시됐다

**single/multi/sub 안내(가볍게):** "Agent 하나면 단독, 여러 개가 이어지면 멀티, 한 Agent가 보조를 부르면 서브예요. 용어는 안 외워도 돼요 — 흐름으로 그리면 자연히 정해져요."

**막힘 해소:** 로드맵이 빈약하면 As-Is 단계를 같이 훑으며 "여기도 Agent가 도울 수 있지 않을까요?" 묻기.

## WS4 · Agent 우선순위 — 표를 대화로 채워 1개 고르기

**목적:** 로드맵의 Agent를 **하나씩 효과·실현가능성으로 따져 표를 채우고**, 그 결과로 1순위 1개를 고른다.
> **중요:** 퍼실리테이터가 표를 *대신 채워 보여주고 확정만 받지 않는다.* 한 Agent씩 **물어서** 표(Agent / 기대 효과 / 실현가능성 / 순위)를 함께 채운다.

**강사 강의 내용:** 효과 × 실현가능성으로 1순위 고르기.

**진행 방식 — 표를 한 행씩:**
1. "로드맵에 Agent가 [A·B·C] 있죠. 하나씩 따져서 표로 정리할게요. 먼저 **[A]** — 이걸 만들면 **효과가 얼마나 커요?** 시간·반복·오류 중 무엇이 얼마나 줄까요? (크다/보통/작다 + 한 줄 근거)"
2. "그리고 **[A]는 만들기 쉬워요?** — 데이터가 이미 있는지 / 규칙·기준이 명확한지 / 외부 연동이 필요한지. (쉽다/보통/어렵다 + 한 줄)"
3. (다음 Agent로) "그럼 **[B]**는요? 효과는? 만들기는?" … **모든 Agent**를 채울 때까지 반복.
4. (표가 다 차면 되비추기) "정리하면 이렇게 돼요 👇 [표를 보여줌]. **효과 크고 만들기 쉬운** 순으로 순위를 매기면 어떻게 될까요?"
5. "그래서 **먼저 만들 1순위 1개**는 무엇으로 할까요? 고른 이유도 한 줄로요."

**프로브:**
- "효과 크다"가 막연하면: "그게 줄여주는 게 *시간*이에요, *반복*이에요, *오류*예요?"
- 우열이 안 갈리면: "효과가 비슷하면, *더 빨리 만들 수 있는*(데이터 있고 연동 적은) 쪽부터예요."

**품질 체크리스트:**
- [ ] **모든 Agent**가 표에 효과·실현가능성과 함께 들어갔다 (1개만 평가 X)
- [ ] 순위가 매겨졌다
- [ ] 1순위 1개 + 고른 이유가 있다

**막힘 해소:** "나머지는 사라지는 게 아니라 다음 차례예요"로 안심. 채운 표는 진행 파일 WS4에 그대로 저장한다.

## WS5 · Agent 정의 — 고른 1개 상세

**목적:** 1순위 Agent를 '글로' 상세 정의한다(역할·작동·Human 개입·연결 도구).

**강사 강의 내용:** 왜 글로 정의하나, 4블록, 트리거 종류, Tool 종류, HITL.

**던질 질문 (블록별, 워크시트 칸을 한 칸씩):**
- 역할: "이 Agent 이름을 붙인다면? / 한 문장으로 뭘 자동화하죠? / 안에서 ①②③ 어떤 순서로 일해요?"
- 작동: "언제 시작돼요? (사람이 누르면 / 파일·메일 오면 / 정해진 시각에 / 앞 Agent가 끝나면) / 끝나면 뭐가 나와요?"
- Human 개입: "Agent가 멈추고 사람이 꼭 확정해야 하는 데는? / 중간에 사람에게 물어 답 받아 다시 하는(HITL) 부분이면 무엇을 물어요?"
- **연결 도구·데이터 — 여기서 선명도가 갈린다. 깊게 판다:**
  - Input: "입력이 정확히 **어떤 파일**이에요?(확장자) 그 안에 **어떤 항목/값**이 들어 있어요? 예: '○○보고서(.pdf)·○○대장(.xlsx) 안에 이름·일자·구분·금액·상태 …' — Agent가 실제로 읽어야 하는 **칼럼/필드**를 짚어주세요." (막연한 "○○ 정보" 금지 → 항목 나열까지)
  - Output: "결과는 **어떤 형태**로, **어떤 항목**을 담아, **어디에** 저장돼요? 예: '대상별 요약 = 핵심 항목 몇 개 → 대시보드/DB 또는 파일'"
  - Tool: "각 단계에서 뭘 써요? (검색 / 문서파싱 / DB / 코드 / 챗)"
- **외부 연동 필요성 검증 (중요):** Tool·데이터에 외부 시스템·API 연동이 끼면 *그냥 받아들이지 말고* 한 번 되묻는다 → "그거 **정말 외부 연동이 꼭 필요해요?** 지금 가진 파일(.md·.pdf·.xlsx)만으로 되는 부분은 아니에요? **연동 없이 되는 범위부터** 잡고, *꼭 필요한 연동만* 남겨요." (외부 연동은 난이도·비용·리스크를 크게 올리므로, 없앨 수 있으면 없앤다. 정말 필요한 경우만 'WS6 구현 방향'으로 넘긴다.)
- 구조: "이 Agent 혼자면 충분해요, 아니면 보조(sub) Agent가 필요해요?"

**품질 체크리스트:**
- [ ] 이름·목적·수행단계
- [ ] 트리거가 4종 중 하나로 분명
- [ ] **Input/Output에 '파일 형식 + 구체 데이터 항목/값'**까지 있다 (막연한 "데이터" X)
- [ ] 외부 연동이 있으면 *정말 필요한지 검증*했다(없앨 수 있는 건 없앴다)
- [ ] Human 개입 지점 명시 (공공·민감 업무는 **HITL 기본**)

**막힘 해소:** 데이터가 막연하면 "그 파일 열면 *맨 윗줄(헤더)에 뭐가 있어요?*"로 항목을 끌어낸다.

## WS6 · PRD 초안

**목적:** 앞 설계를 PRD 9칸으로 정리한다. **핵심 3칸(대상 사용자·기능 요구사항·구현 방향)**을 꼭 채운다.

**강사 강의 내용:** PRD란? (기획안='이걸 하고 싶다' ↔ PRD='이렇게 만든다'), PRD 9칸.

> **중요:** '비는 칸만 자동으로 채우지' 않는다. **9칸을 차례로 하나씩 짚어** 확인·보강한다(앞에서 나왔어도 PRD 관점에서 또렷하게). 특히 7·8·9는 새로 파고든다.

**던질 질문 (9칸을 순서대로, 한 칸씩):**
1. 개요: "이 Agent를 한 줄로 소개하면? (이름 / 뭘 하는지 / 어떤 가치)"
2. **업무 맥락 + Pain 콕 집기:** "As-Is 흐름을 한 줄로. 그리고 **핵심 Pain 3개를 콕 집으면?** 각각 시간/반복/오류/분산 중 무엇인지까지." (뒤의 8번 효과가 여기서 나온다)
3. **대상 사용자:** "이걸 *실제로* 누가, 언제, 어디서 써요? 빈도는?"
4. 선정 이유: "로드맵 중 왜 이걸 1순위로?"
5. **기능 요구사항:** "Agent가 *해야 하는 일*을 기능 단위로 — 트리거·입력·출력 포함."
6. Human 개입 설계: "단독 / HITL / 규제 제약은?"
7. **구현 방향 — 먼저 '무엇으로 만들지'부터 묻는다(필수):**
   - 플랫폼 질문: "이걸 **무엇으로 만들 생각이에요?** ① IDE에서 직접 코딩 ② AI 코딩 어시스턴트 ③ n8n 같은 워크플로우 자동화 툴 ④ Low-code 플랫폼(드래그&드롭) ⑤ 아직 모름 — **이 선택에 따라 구현 방향이 완전히 달라져요.**" (특정 회사·제품을 띄우지 말고 유형으로 안내)
   - 선택을 받은 뒤 거기에 맞춰: "그럼 [선택]에서는 — 데이터를 어떻게 넣고(파일 업로드/DB 연결), 어떤 노드·모듈로 처리하고, 결과를 어디에 둘지"를 구체화한다. (모르면 후보 2개를 장단점으로 비교해 **수강생 수준을 끌어올린다**)
   - 준비 데이터: "미리 갖춰야 할 데이터(항목·형식·출처)는?"
8. **기대효과 & 지표 (2번 Pain에서 도출):** "2번에서 Pain이 *'○○'*였죠 — 이게 줄면 **무엇이 얼마나** 좋아져요? 숫자로요(시간 X→Y, 반복 X회→0). 그걸 **무엇으로 측정**해요(KPI)?" ※ Pain ↔ 효과 ↔ KPI가 한 줄로 이어지게.
9. **리스크 & 전제 (반드시 함께 점검 — 자동 채움 금지):** "도입할 때 걸리는 걸 같이 짚어볼게요 — 데이터 권한·접근, 정확도 한계, 개인정보, 오류 시 사람 재검토, (연동이 있으면) 연동 안정성. 해당되는 거 있어요?"

**품질 체크리스트:**
